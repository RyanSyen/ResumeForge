## Approach
Client-side extraction (`pdfjs-dist` for PDF text layer, `mammoth` for DOCX â†’ plain text), both dynamically `import()`-ed only when the user picks a file so the main bundle is untouched for non-importing users. Extracted text is assembled into a single structured-parse prompt sent to Gemini via a **new** `importResume()` function in `src/lib/gemini.ts` that uses the `x-goog-api-key` header (per CLAUDE.md hard rule and spec constraint) rather than copying the existing `?key=` call â€” existing calls (`improveText`, `generateSummary`, `tailorResume`, `testConnection`) are left untouched since migrating them is out of scope and non-trivial (touches every existing caller/test). The response is validated through F-002's `parseResumeData()` (strict schema, repairs missing/blank ids, rejects wrong-typed fields) so imported data can never bypass the same guarantees JSON import already has. A new `ImportDialog` modal (mirroring `SettingsDialog`'s fixed-overlay/`print:hidden`/backdrop-click pattern) drives the whole flow â€” pick file â†’ extracting â†’ preview-before-apply (section-level summary, replaces `Toolbar`'s bare `setResume()` call) â†’ apply or cancel â€” with a graceful-failure state that shows the raw extracted text for copy/paste instead of a bare error. No merge-into-existing-resume logic (spec: replace only, out of scope) and no OCR/image-PDF handling (out of scope â€” a text-layer-empty PDF is treated as an extraction failure, not silently guessed at).

Rejected alternative: reusing `AiPanel`'s `ApplyCard` component directly â€” its "apply this single field/list" per-field diff model fits AI *tailoring* increments, not a one-shot whole-resume replace with a single confirm/cancel gate (AC3). The plan reuses the *concept* (inline confirm-before-mutate card) but builds a purpose-fit preview summary instead of forcing the import flow through per-field ApplyCards.

Rejected alternative: parsing PDF/DOCX to structured data without an LLM (regex/heuristic section detection) â€” real-world resumes have too much layout variance (single vs. two-column, varying section headers, inconsistent date formats) for a heuristic parser to hit AC1/AC2 reliably; the spec already mandates the Gemini structured-parse step.

## Risk register
| Risk / landmine | Mitigation |
|-----------------|------------|
| `pdfjs-dist` worker file not resolving correctly under Vite (common pain point â€” wrong worker URL causes a silent runtime failure or CORS/MIME error) | Verify with the real PDF fixture in an actual dev-server run (not just jsdom) before considering Step 3 done; jsdom cannot run real Web Workers, so this is a manual-verification gap, not just a unit-test one |
| Gemini hallucination/misparse on unusual resume layouts (two-column, tables, non-standard section headers) not covered by the single fixture | Out of scope per spec (single-column text PDF is the AC1 bar); prompt includes explicit "never invent facts, only structure what's present" instructions matching existing `tailorResume`/`improveText` guardrails; AC3's preview-before-apply is the user-facing safety net for any misparse, not silent trust |

| DOCX table/formatting edge cases (`mammoth`'s plain-text extraction flattens tables, may lose column structure resumes sometimes use for skills) | Accepted risk â€” spec only requires AC2 parity with the *same* resume as the PDF fixture, not arbitrary DOCX layouts; flatten-to-text is mammoth's designed behavior and matches the "extract text â†’ let Gemini structure it" approach used for PDF too |
| New `x-goog-api-key` header call path duplicating request/error-handling logic already in `generateContent()`, risking drift between the two auth patterns | Keep the new path as a minimal, clearly-commented variant colocated in `gemini.ts` next to `generateContent`, reusing the same error-mapping logic (400/403/429 messages) rather than reimplementing it; note in a code comment that this is the intended pattern for all *future* new AI calls per CLAUDE.md, and that migrating existing calls is a follow-up, not part of this feature |

| Test flakiness from real PDF/DOCX fixture parsing in CI (binary fixtures, environment-dependent parsing) | Keep fixtures minimal/deterministic (plain text, no fonts/images); pin `pdfjs-dist`/`mammoth` versions in `package.json` matching the repo's existing `^` convention so CI parsing behavior doesn't drift silently |

## Deviations
- **PDF worker: switched to `pdfjs-dist/legacy/build/pdf.mjs`, and â€” after a wrong turn â€” ended up configuring `workerSrc` conditionally rather than skipping it.** The plan's Step 3 anticipated needing to resolve `pdfjs-dist`'s worker file under Vite as the primary risk; that turned out to be correct, but it took two attempts to get there. The main `pdfjs-dist` browser build requires `DOMMatrix`, which jsdom (the test environment) doesn't provide â€” the library itself warns "Please use the `legacy` build in Node.js environments" â€” so switched to the legacy build. First attempt: left `GlobalWorkerOptions.workerSrc` unconfigured entirely, reasoning that `PDFWorker#initialize`'s `new Worker(...)` call is wrapped in a try/catch that falls back to a main-thread "fake worker." **This passed jsdom tests but threw `No "GlobalWorkerOptions.workerSrc" specified."` uncaught in the real browser preview (Step 15)** â€” the actual source shows `let { workerSrc } = PDFWorker` (a getter that throws when unset) executes *before* the try block starts, so an unset `workerSrc` only fails safely in Node/jsdom, where pdfjs's `isNodeJS` static check disables workers before that getter is ever reached. Real browsers have no such short-circuit. Fix: configure `workerSrc` via Vite's `new URL('pdfjs-dist/legacy/build/pdf.worker.mjs', import.meta.url)` pattern (the plan's original proposal), but only `if (typeof Worker !== 'undefined')` â€” setting it unconditionally broke the Node/jsdom fake-worker path instead, which resolves the workerSrc URL via a plain ESM `import()` that can't fetch the http(s) URL Vite's dev-time transform produces for `import.meta.url` in that environment. The `typeof Worker` guard makes both environments correct: real browsers configure and use a genuine background worker (confirmed via a real dev-server run in Step 15 â€” file picked via browser JS, actual `fetch` to Google's Gemini endpoint observed, real 400 "invalid key" response, extracted text exactly matching the fixture), Node/jsdom skip configuration and rely on `isNodeJS`'s unconditional disable (confirmed via `pdf.test.ts`'s real-fixture assertions). One risk-register item closed for real, not assumed.
- **Added `tsconfig.vitest.json`, a third TS project referenced from the root `tsconfig.json` (not in the original plan).** `tsc -b` (part of `npm run build`) type-checks test files too, and the new fixture-reading tests need `node:fs`, `node:path`, and `__dirname` â€” unavailable under `tsconfig.app.json`'s browser-only `lib`/`types` (correctly so, for actual app code). Rather than polluting the whole app with Node ambient types, added `tsconfig.vitest.json` (extends `tsconfig.app.json`, adds `"node"` to `types`, scoped via `include` to `src/**/*.test.ts(x)` only) and excluded test files from `tsconfig.app.json` itself so each file is type-checked exactly once, under the config matching its actual runtime environment.
- **Added `withDefaults()` in `gemini.ts` (not in the original plan).** `parseResumeData`'s strict schema requires every string field present on every item â€” correct for validating a user's own JSON export, but Gemini's response won't reliably include every key for every item (e.g. a remote job with no `location`). A first pass at `importResume` without this hit `SchemaError: Invalid "experience": expected an array` on a response that only omitted an optional-seeming field. `withDefaults()` recursively fills genuinely *missing* keys from the same schema-template object already used to build the prompt (single source of truth, no drift), but leaves present-but-wrong-typed values untouched so a real malformed response (e.g. `skills` returned as a string) still fails validation and surfaces as an AI-parsing error rather than being silently coerced away. Covered by `gemini.test.ts`'s "repairs a response missing item ids" and "throws a specific error when the response has a wrong-typed field" cases.
- **`mammoth.extractRawText` called with both `arrayBuffer` and `buffer` fields.** mammoth ships two build variants selected via package.json's `browser` field remap (`lib/unzip.js` reads `options.buffer`, a Node `Buffer`; `browser/unzip.js` reads `options.arrayBuffer`). Vite's client build resolves the browser variant; Vitest (running in a plain Node process) resolves the Node variant. Passing both fields (`{ arrayBuffer, buffer: Buffer.from(arrayBuffer) }` when the `Buffer` global exists, else just `{ arrayBuffer }`) makes `extractDocxText` work correctly under whichever variant gets resolved, without needing custom Vite/Vitest resolution config. Confirmed working against the real DOCX fixture in `docx.test.ts`.
- **AC1/AC2 real-Gemini-key verification explicitly skipped, by user decision (2026-07-03).** No Gemini API key was available in this environment. Options discussed: (a) skip and rely on existing unit tests (schema repair, error handling, defaults-filling â€” all mocked-response-based), noting the gap; (b) the user types a real key directly into the running app's Settings dialog themselves during a live preview session, so the key never passes through the assistant's context/logs, matching the app's own privacy promise. User chose (a), skip. **Residual risk carried forward, not resolved**: whether Gemini's actual output quality for a typical real-world resume satisfies AC1 ("correctly populated sections") and AC2 (DOCX parity) is unverified â€” this is a prompt-quality question, not a code-correctness one, and needs a human with a real key to check before Gate 2. Everything downstream of a Gemini response (schema validation/repair, id assignment, preview rendering, apply/cancel) is verified via `gemini.test.ts`/`ImportDialog.test.tsx`'s mocked-response cases.

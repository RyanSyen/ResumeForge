# F-005 — Implementation Plan

*Status: approved · Approved by: Ryan Wong · Date: 2026-07-03*

## Approach

Keep `SectionKey` as the fixed 7-key literal union unchanged (every built-in call site — store actions, `SECTION_LABELS`, `sectionBody()` — keeps working as-is with zero ripple). Add `CustomSection`/`CustomSectionItem` types and `customSections: CustomSection[]` on `ResumeData`, and widen only the two order/visibility fields plus `moveSection`/`toggleSection` to a new `type SectionId = SectionKey | string`, where a custom section's id (a `newId()` UUID) slots into `sectionOrder`/`hiddenSections` exactly like a built-in key. Rendering resolves a `SectionId` to a label/body via a small dispatch helper (`sectionLabel`, `sectionBody`) that checks the built-in map first and falls back to looking up the matching `CustomSection` — this replaces the exhaustive `switch`/`Record<SectionKey, …>` patterns in `EditorPanel.tsx` and `templates.tsx` with an "is it a built-in key or a custom id" branch, which is the smallest change that satisfies "participate exactly like built-ins."

Alternatives rejected: (a) making `SectionKey` itself `string`-based / opening the union — rejected, it would drop type safety on every built-in call site (`resume.experience`, `resume.skills`, etc. all key off literal `SectionKey`) for no benefit, since only order/visibility need to hold foreign ids; (b) a fully generic "sections registry" refactor (data-driven config replacing all per-section editors/templates) — rejected as out-of-scope scope creep; F-005 only needs custom sections to append onto the existing hardcoded 7, not a rewrite of how built-ins work.

3. [x] **Store — section-level actions**: In `source_code/src/store/resume.ts`, widen `moveSection`/`toggleSection` params to `SectionId` (implementation bodies are already generic — `order.indexOf(key)` / `hiddenSections.includes(key)` work unchanged on strings). Add `addCustomSection(title: string): void` (creates `{ id: newId(), title, items: [] }`, appends to `customSections`, appends id to `sectionOrder`), `renameCustomSection(id: string, title: string): void`, `removeCustomSection(id: string): void` (removes from `customSections`, and filters the id out of both `sectionOrder` and `hiddenSections` in the same `set()` call — this is the one place AC4's "without corrupting either" guarantee is enforced). 4. [x] **Store — item-level actions for custom sections**: Add 4 parallel actions scoped to custom items — `addCustomItem(sectionId, item)`, `updateCustomItem(sectionId, itemId, patch)`, `removeCustomItem(sectionId, itemId)`, `moveCustomItem(sectionId, itemId, dir)` — each doing the same immutable map/filter/swap logic as the existing built-in actions but reaching into `s.resume.customSections.find(cs => cs.id === sectionId).items`. ## Risk register

| Risk / landmine | Mitigation |
|-----------------|------------|
| Type ripple from loosening `sectionOrder`/`hiddenSections` typing spreads further than expected (`resume.ts`, `EditorPanel.tsx`, `templates.tsx`) | `SectionKey` itself is left untouched; only `SectionId = SectionKey \| string` is introduced for the 2 order/visibility fields + `moveSection`/`toggleSection`. Step 1 is a dedicated "build and observe all type errors" step before any fix lands, confirming the ripple surface empirically. |
| Schema validation is too shallow — a `sectionOrder`/`hiddenSections` entry could reference a custom section id that doesn't exist in `customSections` (forged/stale id), corrupting rendering silently | Step 6 adds a cross-field Zod check (`superRefine`) so strict `parseResumeData` (used on import) rejects dangling ids outright; `repairResumeData` (lenient, used for localStorage rehydration) drops unknown ids rather than crashing. Step 7 adds an explicit test for the rejection case. |
| Print/pagination regression (F-003 multi-page handling) when a new section type is added to templates | `CustomSectionBody` reuses the exact `print-avoid-break` / `print-avoid-break-after` classes and item-wrapper idiom used by `ExperienceBody`/`ProjectsBody`, not a bespoke layout. Step 11's smoke test asserts these classes are present; manual verification includes printing/paginating a resume with a custom section long enough to span pages. |
| Editor's hardcoded per-section switch — extending it awkwardly (cramming custom sections into the same `Record<SectionKey, ReactNode>`) would require fake `SectionKey` values or unsafe casts | Steps 8-9 keep the built-in `Record<SectionKey, ReactNode>` untouched, dispatching custom ids through a separate `CustomSectionEditor` via an `if/else` branch keyed on "is this id a known `SectionKey`," avoiding any cast or union hack. |
| Delete-with-confirmation for a custom section might diverge from the app's existing `confirm()` convention, introducing a second UX pattern for destructive actions | Step 8 reuses the exact `confirm(...)` call pattern from `Toolbar.tsx:76`, not a new modal/dialog component. |
| Migration bump (v1→v2) forgotten or mis-tested, silently losing existing users' resumes on the next load | Step 5 is a standalone, independently-verifiable step with its own dedicated test (payload with no `customSections` key migrates cleanly; all other fields byte-identical) before any other step depends on it — matches CLAUDE.md's mandatory-migrate-bump-with-test rule. |
| `resumeToText()` custom-section serialization balloons prompt size or leaks malformed data into the Gemini prompt if a custom section has huge item lists | Out of scope to bound/truncate for this feature (existing built-in sections have no such guard either); noted as a non-blocking observation, not a required mitigation for F-005. |

## Test plan

**Unit — store** (`source_code/src/store/resume.test.ts`):
- `addCustomSection` creates a section with a generated id, appends to `customSections` and `sectionOrder`.
- `renameCustomSection` patches only the `title` of the matching section.
- `removeCustomSection` removes the section from `customSections` AND strips its id from both `sectionOrder` and `hiddenSections` in one update (AC4).
- `addCustomItem`/`updateCustomItem`/`removeCustomItem`/`moveCustomItem` — append, patch-preserves-other-fields, no-op-on-missing-id, remove-only-matching, swap-middle, no-op-first-up/last-down — mirroring existing `addItem`/`updateItem`/`removeItem`/`moveItem` test shapes.
- `moveSection`/`toggleSection` work when passed a custom section id (not just a `SectionKey`).
- `migrateResumeState`: v1 payload without `customSections` → v2 result has `customSections: []`, all else unchanged; v0 payload → still produces a fully valid v2 resume via the repair path.

**Unit — schema** (`source_code/src/lib/schema.test.ts`):
- `parseResumeData` round-trips a resume with populated `customSections` unchanged.
- `parseResumeData` repairs missing item ids inside a custom section's items.
- `parseResumeData` throws `SchemaError` on wrong-typed `customSections`.
- `parseResumeData` throws `SchemaError` when `sectionOrder`/`hiddenSections` contains an id that isn't a built-in key and doesn't match any `customSections` entry.
- `repairResumeData` never throws on garbage `customSections` input, falls back to `[]`.

**Integration/UI**:
- `EditorPanel`/`CustomSectionEditor`: add a custom section, rename it, add/edit/reorder/delete items, delete the section (confirm dialog stubbed/mocked), verify store state after each action.
- Template smoke test (new or extended `templates.test.tsx`/`Preview.test.tsx`): resume with one custom section renders its title and item text in `ModernTemplate`, `ClassicTemplate`, `CompactTemplate`.
- Hide/reorder integration: toggling a custom section's visibility removes it from rendered output; moving it up/down changes render order — same assertion style as existing `moveSection`/`toggleSection` store tests, extended to a rendered-template check.

**Manual verification script** (for the review stage — walk each acceptance criterion by hand):
1. Add a custom section named "Publications," add 2 items with title/subtitle/date/description/bullets, reorder them, delete one — confirm the preview updates live across all 3 templates (switch template selector and check each).
2. Move the custom section up/down in the section order; toggle hide/show — confirm it reorders and disappears/reappears in the preview exactly like a built-in section (e.g. Skills).
3. Export to JSON, reload the app (or clear state), import the JSON back — confirm the custom section and its items round-trip losslessly. Separately: reload the page without exporting (localStorage persistence) — confirm the resume survives with `customSections` intact (migration path).
4. Delete the custom section — confirm a confirmation prompt appears, cancelling leaves it intact, confirming removes it and it no longer appears in section-order/hide controls (no orphaned entries, no console errors).
5. With a custom section present, run AI Tailor against a job description — confirm no error occurs and the custom section's content is present in the request context, while confirming bullet-rewrite suggestions still only target Experience items.

# Living Product Map & Feature Inventory

*Last updated: 2026-07-02 ‚Äî generated from `source_code/src` as-is. Update this file whenever a feature's behavior changes.*

## 1. System Overview & Conventions

*   **Tech Stack:** Vite 8 + React 19 + TypeScript 6 SPA ("ResumeForge"). Tailwind CSS v4 (via `@tailwindcss/vite`), Zustand 5 for state, `lucide-react` icons, `oxlint` for linting. **No backend, no database, no router, no auth** ‚Äî the entire app is client-side; all persistence is browser `localStorage`.
*   **AI Provider:** Google Gemini REST API (`https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent`), called **directly from the browser** with a user-supplied API key (BYO-key model). Selectable models: `gemini-2.5-flash` (default), `gemini-2.5-pro`, `gemini-2.5-flash-lite`.
*   **State Management:** Three persisted Zustand stores (all use `persist` middleware ‚Üí localStorage):
    *   `resume-builder:resume` (`src/store/resume.ts`) ‚Äî the entire `ResumeData` object + all mutation actions (`setBasics`, `addItem`, `updateItem`, `removeItem`, `moveItem`, `moveSection`, `toggleSection`, `loadSample`, `reset`).
    *   `resume-builder:settings` (`src/store/settings.ts`) ‚Äî Gemini `apiKey`, `model`, `template`, `accent`. `settingsOpen` (modal visibility) is excluded from persistence via `partialize`.
    *   `resume-builder:ai` (defined inline in `src/components/ai/AiPanel.tsx`) ‚Äî last pasted job description + last `TailorResult`.
*   **Data Model:** Single `ResumeData` type in `src/types.ts` ‚Äî `basics` + 7 sections (`summary`, `experience`, `education`, `projects`, `skills`, `certifications`, `languages`) + `sectionOrder` + `hiddenSections`. All list items carry a stable `crypto.randomUUID()` `id` (generated by `newId()` in `src/store/resume.ts`).
*   **Global Layout** (`src/App.tsx`): Single screen, no routes.
    *   Top: `Toolbar` (branding, template select, accent swatches, Sample/Reset, Import/Export JSON, Download PDF, settings gear).
    *   Left: fixed 460px sidebar with two tabs ‚Äî **Content** (`EditorPanel`) and **AI Tailor** (`AiPanel`).
    *   Right: `Preview` ‚Äî live A4 page render of the resume.
    *   Overlay: `SettingsDialog` modal (visibility driven by settings store).
    *   Everything except the preview page carries `print:hidden` ‚Äî printing is the PDF export mechanism (see Feature: PDF Export).

## 2. Core Feature Inventory

### Feature: Resume Content Editor
*   **User Value:** Structured, form-based editing of every resume section with instant live preview; sections and items can be reordered, hidden, or deleted without losing data.
*   **Existing UI/UX States:**
    *   Empty State: Each section shows a dashed "Add ‚Ä¶" button (`AddButton`); `reset()` produces an empty resume, `loadSample()` loads a full demo resume. First load ships with the sample resume.
    *   Active State: `BasicsEditor` grid of 8 fields; each list section renders collapsible `<details>` item cards (`ItemCard`) with move-up/move-down/delete icon buttons. Section headers (`SectionShell`) have move-up/move-down and an eye toggle; hidden sections render at 60% opacity with their body collapsed.
    *   Loading/Success/Error States: No async states ‚Äî every keystroke writes synchronously to the Zustand store and persists to localStorage immediately.
*   **Frontend Components:** `src/components/editor/EditorPanel.tsx` (per-section editors), `src/components/editor/SectionShell.tsx` (`SectionShell`, `ItemCard`), `src/components/ui.tsx` (`Field`, `TextArea`, `Button`, `IconButton`).
*   **Backend & Data Layer:** None. Reads/writes `resume-builder:resume` in localStorage via the Zustand persist middleware. No network calls.
*   **Known Edge Cases Handled:**
    *   Experience/project highlights are edited as one textarea, split on `\n` (`splitLines`); empty lines are preserved while typing.
    *   Skills are comma-separated; splitting uses `trimStart()` only, so trailing spaces inside a token survive.
    *   Item/section move actions are bounds-checked (no-op at first/last position); buttons are disabled at the edges.
    *   `Reset` requires a native `confirm()`; there is no undo.

### Feature: Inline AI Writing Assist (Improve / Generate)
*   **User Value:** One-click AI rewrite of the professional summary or experience bullets, and generation of a summary from the full resume ‚Äî without ever silently overwriting the user's text.
*   **Existing UI/UX States:**
    *   Empty State: "Improve with AI" is disabled when the target text is empty; "Generate from resume" is always enabled.
    *   Active State: Violet sparkle button per assist point.
    *   Loading/Success/Error States: Button shows spinner + "Thinking‚Ä¶" while in flight; result appears as an inline violet suggestion card with **Apply** / **Discard**; errors render as inline red text. Applying experience bullets filters out empty lines.
*   **Frontend Components:** `AiAssist` in `src/components/ui.tsx`; wired up in `SummaryEditor` and `ExperienceEditor` inside `src/components/editor/EditorPanel.tsx`.
*   **Backend & Data Layer:** `improveText()` / `generateSummary()` in `src/lib/gemini.ts` ‚Üí Gemini `generateContent` (temperature 0.4). Prompts instruct the model to never invent facts and to preserve line count/structure. No triggers or background jobs.
*   **Known Edge Cases Handled:** Missing API key throws `MissingKeyError` with a settings-pointing message; empty Gemini responses throw a retryable error; suggestion state is local component state (lost on tab switch ‚Äî intentional, apply-or-discard flow).

### Feature: AI Job Tailoring (AI Tailor tab)
*   **User Value:** Paste a job description ‚Üí get an ATS match score, matched/missing keywords, a tailored summary, implied skills to add, rewritten experience bullets, and recommendations ‚Äî each applied only with explicit user approval.
*   **Existing UI/UX States:**
    *   Empty State: If no API key, an amber "Gemini API key required" banner with an "Add API key" button (opens settings). The analyze button is disabled until the JD is ‚â• 40 trimmed characters (a hint asks for a fuller JD below that).
    *   Active State: JD textarea + "Analyze & tailor resume" button; results render as a match-score card (color-coded ‚â•75 green / ‚â•50 amber / else red, with progress bar), keyword chips, and per-suggestion `ApplyCard`s.
    *   Loading/Success/Error States: Button shows spinner + "Analyzing with Gemini‚Ä¶"; API errors render in a red inline alert; each `ApplyCard` flips to a green "Applied" check once its content matches the resume.
*   **Frontend Components:** `src/components/ai/AiPanel.tsx` (`AiPanel`, `TailorResults`, `ApplyCard`, `SummarySuggestion`, `SkillsSuggestion`, `ExperienceSuggestions`, plus the inline `useAi` store).
*   **Backend & Data Layer:** `tailorResume()` in `src/lib/gemini.ts` ‚Äî single Gemini call with `responseMimeType: application/json`, temperature 0.3, strict JSON schema in the prompt. The resume is serialized via `resumeToText()`, with experience items carrying their stable `id`s so bullet rewrites map back deterministically. Response is defensively normalized (score clamped 0‚Äì100, arrays type-checked, experience suggestions filtered to valid `{id, highlights[]}`).
*   **Known Edge Cases Handled:**
    *   `parseJson()` strips markdown code fences and falls back to slicing the outermost `{...}` if the model wraps JSON in prose.
    *   Experience suggestions whose `id` no longer exists in the resume (user deleted the item) are silently dropped.
    *   Skills-to-add are deduplicated case-insensitively against existing skills; applied to the first skill group, or a new "Skills" group is created if none exists.
    *   "Applied" detection is equality-based (summary string match, `JSON.stringify` of highlights), so later manual edits flip a card back to un-applied.
    *   JD and result persist in `resume-builder:ai` ‚Äî surviving reloads and tab switches.

### Feature: Live Preview & Templates
*   **User Value:** Real-time A4 rendering of the resume with three switchable templates and six accent colors, so the user always sees exactly what will export.
*   **Existing UI/UX States:**
    *   Empty State: Renders the (mostly blank) A4 page; hidden sections and empty sections simply don't render content.
    *   Active State: Template select (`modern` / `classic` / `compact`) and accent swatches in the toolbar; floating zoom pill (bottom-right) 40%‚Äì150% in 10% steps, default 85%.
    *   Loading/Success/Error States: None ‚Äî synchronous re-render from store state. Unknown template id falls back to `ModernTemplate`.
*   **Frontend Components:** `src/components/preview/Preview.tsx`, `src/components/preview/templates.tsx` (the three template components), template/accent controls in `src/components/Toolbar.tsx`.
*   **Backend & Data Layer:** None. Reads resume store + settings store (`template`, `accent`).
*   **Known Edge Cases Handled:** Zoom uses the CSS `zoom` property on `#preview-zoom` and is forced back to 1 for printing. Page is fixed at `210mm √ó min-height 297mm` ‚Äî content longer than one page overflows naturally with **no pagination controls or page-break handling**.

### Feature: PDF Export (Print-to-PDF)
*   **User Value:** Download the resume as a pixel-accurate A4 PDF.
*   **Existing UI/UX States:** Single "Download PDF" toolbar button ‚Üí opens the browser's native print dialog (`window.print()`); user chooses "Save as PDF". No in-app loading/success/error states.
*   **Frontend Components:** Button in `src/components/Toolbar.tsx`; the actual mechanism is the `@media print` block in `src/index.css`.
*   **Backend & Data Layer:** None ‚Äî fully browser-native. No PDF library.
*   **Known Edge Cases Handled:** Print CSS hides everything (`body * { visibility: hidden }`) then re-shows only `#resume-page`, absolutely positioned at the page origin; `@page { size: A4; margin: 0 }`; preview zoom and box-shadow are reset. **Fragile by design** ‚Äî see Technical Debt.

### Feature: JSON Import / Export (Data Portability)
*   **User Value:** Full resume backup/restore and portability as a JSON file; the de-facto "save file" since there is no cloud storage.
*   **Existing UI/UX States:** "Export" downloads `<full-name-kebab>.json` (falls back to `resume.json`); "Import" opens a file picker (accepts `application/json`) and replaces the current resume. Import failure shows a native `alert()` with the error message.
*   **Frontend Components:** Buttons + hidden file input in `src/components/Toolbar.tsx`; logic in `src/lib/file.ts` (`exportResumeJson`, `importResumeJson`).
*   **Backend & Data Layer:** None ‚Äî `Blob` + object URL for export, `FileReader` for import.
*   **Known Edge Cases Handled:** Import validates that the parsed value is an object with a `basics` key, then spreads it over `emptyResume()` so missing top-level fields get defaults; `sectionOrder` falls back to the default order if absent/empty; the file input value is cleared after each pick so re-importing the same file works. Validation is shallow ‚Äî see Technical Debt.

### Feature: AI Settings & Key Management
*   **User Value:** Bring-your-own free Gemini API key, choose a model, and verify connectivity ‚Äî with a clear privacy promise (key never leaves the browser except to Google).
*   **Existing UI/UX States:**
    *   Empty State: Amber dot on the toolbar gear when no key is set; AI features route the user here (`openSettings()`) instead of failing.
    *   Active State: Modal with password-masked key input (show/hide eye toggle), link to `aistudio.google.com/apikey`, model dropdown.
    *   Loading/Success/Error States: "Test connection" button ‚Äî spinner while testing, green check + "Connected" on success, red error message on failure. Editing the key resets test state to idle.
*   **Frontend Components:** `src/components/SettingsDialog.tsx`; store in `src/store/settings.ts`; `testConnection()` in `src/lib/gemini.ts`.
*   **Backend & Data Layer:** Key + model persist to `resume-builder:settings` in localStorage. Test connection sends a minimal one-word prompt through the same `generateContent` path.
*   **Known Edge Cases Handled:** Backdrop click closes the modal (inner panel stops propagation); Gemini error mapping in `src/lib/gemini.ts` ‚Äî 400 with "api key" in the message ‚Üí "invalid key", 403 ‚Üí "key rejected", 429 ‚Üí "rate limit reached, wait a moment", other ‚Üí server detail or generic HTTP message.

## 3. Current System Constraints & Safeguards

*   **Auth Rules:** There is **no authentication and no protected routes** ‚Äî the app is a single-page, single-user, local-only tool. All user data (resume, JD, AI results, API key) lives in the browser's localStorage under the `resume-builder:*` keys. Clearing site data destroys everything; JSON export is the only backup path. Any feature spec assuming "logged-in user" or "past resumes dashboard" does **not** exist yet (it's roadmap in the SSOT doc, not shipped).
*   **Rate Limits / AI Costs:** Zero platform-side AI cost ‚Äî every Gemini call is billed to the user's own key. There is **no app-level rate limiting, quota, debounce, or paywall**; the only friction is: per-control single-in-flight (buttons disabled while loading), the 40-character minimum JD length, and Google's own per-key quotas (429s are caught and surfaced as a friendly retry message).
*   **Technical Debt Notes (be careful around these):**
    1.  **API key exposure:** the key is stored in plaintext localStorage and sent as a URL query param (`?key=`) in `src/lib/gemini.ts` ‚Äî it appears in browser network logs and any intermediary that logs URLs. Moving to the `x-goog-api-key` header (or a backend proxy) is the eventual fix; don't copy the query-param pattern into new AI calls.
    2.  **PDF export is a print-CSS hack** (`src/index.css`): `visibility: hidden` on `body *` + absolute positioning of `#resume-page`. Any new fixed/portal UI (toasts, modals) must carry `print:hidden` or it can bleed into the PDF. CSS `zoom` is used for preview scaling (non-standard; behaves differently in Firefox). No multi-page/page-break handling ‚Äî long resumes overflow page 1 silently.
    3.  **Shallow import validation** (`src/lib/file.ts`): only `basics` presence is checked; list items are not schema-validated, so a malformed JSON with e.g. `experience: [{}]` (missing `id`s) will import and can break item keying, move/remove actions, and AI tailoring id-matching.
    4.  **Native `alert()`/`confirm()`** are used for import errors and reset confirmation (`src/components/Toolbar.tsx`) ‚Äî replace with in-app UI before polishing, but don't remove the reset confirmation.
    5.  **"Applied" detection is value-equality** (`AiPanel.tsx`): summary via string compare, bullets via `JSON.stringify` ‚Äî refactors that reorder or normalize these values will silently break the Applied indicators.
    6.  **`useAi` store is defined inside a component file** (`src/components/ai/AiPanel.tsx`), unlike the other stores in `src/store/` ‚Äî move it if it grows.
    7.  **Skills apply targets the first skill group arbitrarily** (`SkillsSuggestion`), and skill splitting uses `trimStart()` only.
    8.  **No tests of any kind** ‚Äî nothing guards the store mutations, Gemini JSON parsing fallback (`parseJson` brace-slicing), or import merge logic. These are the highest-risk areas to touch blind.
    9.  **Persist-schema versioning is absent:** none of the three Zustand stores declare a `version`/`migrate` ‚Äî changing `ResumeData` or settings shape will hydrate stale localStorage into the new shape unchecked.
